## 오늘 배운 것

"이 에이전트는 대화할수록 스스로 똑똑해진다"는 말의 실체:
- **LLM 모델 자체는 기억이 없다(stateless).** 매 호출 백지에서 시작한다.
- 기억은 **하네스(프레임워크)** 가 한다 — 대화를 저장해뒀다가 다음 호출 프롬프트에 다시 넣어주는 방식.
- 그래서 "기억"은 모델 능력이 아니라 **배관(plumbing)** 이다. 어떤 모델에든 붙일 수 있다.

오픈소스 [hermes-agent](https://github.com/NousResearch/hermes-agent)를 로컬에 격리해서 직접 띄워 이 과정을 눈으로 확인했고, Claude Code와 비교해봤다.

## 1. 모델 vs 하네스

```
LLM 모델          = CPU.  기억 없음. 매번 백지.
메모리 DB/파일     = 디스크. 상태 저장.
하네스            = OS.  둘을 이어줌.
```

모델은 절대 안 변한다. 하네스가 "지난 걸 다시 프롬프트에 넣어주는" 방식으로 기억하는 *척*하게 만든다. 그래서 같은 모델이라도 하네스가 좋으면 더 똑똑해 보인다.

## 2. hermes의 2층 메모리

hermes를 빈 작업 디렉터리로 띄워 "나는 Demo, 땅콩 알레르기 있어" 같은 대화를 몇 턴 했더니, 디렉터리에 두 종류의 기억이 생겼다.

**아랫층 — SQLite (`state.db`)**: 대화 원본이 통째로 들어간다.
- `messages` 테이블 = 모든 발화(역할·내용·토큰·도구호출)
- `messages_fts_trigram` = **전문검색(FTS5) 색인** → 자기 과거 대화를 키워드로 검색 가능
- `sessions` 테이블 = 세션별 모델·토큰·비용까지 (단순 메모리가 아니라 자기 관측 DB)

실제로 이 DB에서 `땅콩`을 검색하니 관련 메시지가 다 나왔다. 에이전트가 자기 기억을 검색하는 1급 기능이 내장돼 있는 것.

**윗층 — 마크다운 (`USER.md` / `MEMORY.md`)**: 원본에서 *증류한 한 줄 요약*.
```
이름: Demo. 한국어로 결론부터 짧게. 땅콩 알레르기 있음.
§
사이드프로젝트: Foobar (Rust CLI 도구).
```
핵심은 "많이"가 아니라 "정수만" — 글자 수 한도(사용자 프로필 ~1.4KB)가 일부러 작다. raw 대화를 다 쌓지 않고 압축한다.

**결정적 확인**: 대화 맥락이 0인 *완전히 새 세션*에서 "내가 누구지?"라고 물으니 USER.md만으로 다 기억했다. 이게 "세션을 넘어 똑똑해진다"의 실체다.

흐름을 한 줄로: `대화 → SQLite에 원본 적재 → 증류 → USER.md → 다음 세션에 주입`

## 3. Claude Code는 SQLite를 쓸까?

안 쓴다. `~/.claude` 전체를 뒤져도 `.db` 파일이 하나도 없다. 대신:

| 층 | hermes | Claude Code |
|---|---|---|
| 요약·선별 | `USER.md`/`MEMORY.md` 마크다운 | `MEMORY.md`+사실별 `.md` 마크다운 (**같음**) |
| 대화 원본 | `state.db` **SQLite** + FTS 색인 | 세션별 **`.jsonl`** (append-only 로그) |
| 과거대화 검색 | DB 내장 FTS로 빠름 | 로그라 색인 없음 → grep/외부도구 |

즉 **둘 다 "원본 저장 + 요약 증류" 2층인데, hermes는 DB파, Claude Code는 파일파.** SQLite를 안 쓴다고 기억을 못 하는 게 아니라, 파일(JSONL)로 하는 것뿐이다.

왜 갈렸나:
- hermes = 서버에 상주하는 자율 에이전트 → 과거를 *질의*하고 비용까지 추적해야 함 → 구조화된 DB가 맞음
- Claude Code = 프로젝트별 CLI 개발 도구 → 사람이 읽고 git diff로 남고 세션을 그대로 재생 → 단순한 파일이 맞음

## 4. 삽질에서 배운 gotcha 2개

hermes에서 메모리가 안 써져서 한참 헤맸는데, 원인이 둘이었다 (재현하려는 사람을 위해):

1. **메모리 도구는 toolset을 켜야 뜬다.** 기본 toolset만으론 `memory` 도구가 안 붙는다. `hermes chat -t hermes-cli,memory`처럼 명시해야 메모리 쓰기가 작동한다.
2. **`--ignore-rules`가 메모리를 꺼버린다.** 이 플래그가 내부적으로 `skip_memory=True`로 매핑돼서(소스 `main.py`), 메모리 저장소가 아예 생성되지 않는다. 그러면 봇이 "메모리가 비활성화돼 있다"고 답한다. 메모리를 관찰하려면 이 플래그를 빼야 한다.

플래그 하나가 핵심 기능을 끄는데 에러도 안 나고 조용히 비활성화되니, 소스를 따라가야 원인이 보였다.

## 결론

"self-improving AI"는 마법이 아니라 **잘 조립한 하네스**다. 저장하고(SQLite/JSONL), 증류하고(마크다운 요약), 다시 주입하는 — 그 배관의 품질이 "똑똑해짐"을 만든다. 모델 등급과는 별개다.

그래서 도구를 평가할 때 "메모리 되냐?"는 변별력이 약하다(다 된다, 하네스 레벨이니까). 진짜 따질 건 **무엇을 기억하고 무엇을 버리나(큐레이션 정확도)** — 틀린 걸 기억하면 이후 모든 대화가 오염되기 때문이다.

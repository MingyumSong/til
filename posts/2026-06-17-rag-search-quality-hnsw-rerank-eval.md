## 오늘 배운 것

검색/RAG 시스템 품질을 올리는 레버들을 정리하다, 정작 가장 중요한 건 레버가 아니라 "잴 자(尺)"라는 걸 깨달았다.

### 0. 먼저 — "semantic"은 두 가지 다른 뜻이다

헷갈려서 짚고 간다.

- **BI의 semantic layer**: 테이블·컬럼에 비즈니스 의미(지표 정의, 차원)를 입혀 사람이 이해하게 매핑하는 *메타데이터 계층*. (dbt, Cube, LookML 류)
- **RAG의 semantic search**: 텍스트를 임베딩 벡터로 바꿔 *의미적 유사도*로 문서를 찾는 *벡터 검색*.

둘 다 "의미"를 다루지만 — 전자는 **정의·매핑**, 후자는 **벡터 거리 기반 검색**이다. 완전히 다른 개념. 아래는 후자(벡터 검색) 이야기.

### 1. HNSW — 층마다 하는 일이 다르다

벡터 검색은 글을 공간 속 점(임베딩)으로 바꾸고, 질문 점에서 *가장 가까운 점들*을 찾는 일이다. 순진하게 하면 수백만 점과 전부 거리 재기 → 너무 느리다. 그래서 **HNSW**(Hierarchical Navigable Small World)라는 다층 그래프 인덱스를 쓴다.

```
위층   ●        ●           ← 점 적음, 링크 길다(급행)
          \    /
중간층 ●—●—●—●—●            ← 중간 밀도
       |  |  |  |
0층    ●●●●●●●●●●●●●●●●●●●   ← 모든 점, 빽빽, 링크 짧다(완행)
```

- **위층·중간층 (coarse search):** 매 단계 이웃 중 질문에 *가장 가까운 1개만* 따라가는 탐욕적 이동. 링크가 길어 공간을 성큼성큼 가로질러 "대충 그 동네"까지 빠르게 내려온다. 후보를 1개만 들고 다닌다.
- **0층 (정밀 확정):** 여기서만 후보를 여러 개 동시에 유지하는 beam 탐색을 한다. 빽빽한 동네에서 탐욕 1개만 쫓다 *지역 최소점*(살짝 잘못된 골목)에 갇히는 걸 막고, 진짜 최근접 top-k를 확정한다.

이름 그대로 **H**ierarchical(층 나눔) + **N**avigable **S**mall **W**orld(친구→친구로 몇 홉 만에 닿는 그래프). "위에서 길찾기, 0층에서 정밀 확정"으로 역할이 갈린다.

### 2. ef_search — recall과 속도를 맞바꾸는 손잡이

0층 beam에서 **후보를 몇 개까지 들고 탐색할지**가 `ef_search`다.

- 크게 → 더 많이 살핌 → **recall↑**(진짜 최근접 놓칠 확률↓), 대신 **느림**
- 작게 → **빠름**, 대신 가끔 진짜 1등을 놓침

핵심: **인덱스를 다시 빌드할 필요 없이 *쿼리 시점*에 바꿀 수 있다.** 인덱스 *빌드* 파라미터(`m`, `ef_construction`)와는 별개. 그래서 튜닝 비용이 거의 공짜다. (단 뽑을 개수 top-k보다 충분히 커야 한다.)

### 3. reranking vs recall — 다른 단계, 다른 문제

검색 품질을 올리는 두 레버는 파이프라인의 *다른 자리*에 있다.

```
질문 → [검색(retrieval)] → [reranking] → top-k → 답변 생성
            ↑                   ↑
       ef_search 여기       rerank 여기
       "놓치지 않기"        "정답을 맨 위로"
       = recall            = precision/순서
```

- **ef_search(recall):** 그물을 얼마나 넓게 던지나 — 정답 문서가 후보에 들어오긴 했나.
- **reranking(precision):** 잡은 후보를 더 정교한 모델로 재정렬 — 정답을 1등으로 올렸나.

원리: **reranking은 "검색이 이미 건진 것"만 재정렬한다.** 정답이 후보에 아예 안 들어오면(recall 실패) 리랭커도 못 살린다. 그래서 *recall이 먼저 충분*해야 reranking이 의미가 있다. 품질이 목표면 보통 reranking이 가장 큰 단일 레버고, recall 튜닝은 그 앞단 곁다리다.

**어떻게 1차 검색과 다른가 (bi-encoder vs cross-encoder):** 1차 검색은 질문과 문서를 *따로* 벡터로 만들어 비교한다(bi/dual-encoder) — 문서 벡터를 미리 계산해 색인할 수 있어 **빠르지만 거칠다**. 리랭커는 질문과 문서를 *한 모델에 같이 넣어* 토큰 수준으로 관련성을 따진다(cross-encoder) — 쿼리마다 계산해야 해 **느리지만 정밀하다**. 그래서 1차로 좁힌 상위 20~25개에만 리랭커를 돌린다. 비유하면 1차 = 서류 키워드 스크리닝, 리랭커 = 추린 후보 면접.

### 4. 키워드 + 의미를 합치기 (하이브리드 검색)

벡터(의미) 검색만으로는 부족할 때가 많다. 고유명사·ID·정확히 일치해야 하는 단어는 오히려 *키워드 검색*이 강하다. 그래서 실무에선 둘을 합친다.

- **키워드(lexical) 검색** — 두 갈래:
  - **전문검색(FTS, full-text search)**: 글을 *단어(토큰)* 단위로 쪼개 매칭. 정확한 단어엔 강하지만, 단어가 안 맞으면 0건.
  - **트라이그램(trigram)**: 글자를 *3개씩* 쪼개 부분일치·오타·변형을 잡는다. 단어 경계가 애매한 언어(한국어 등)나 오타에 강한 *보조 그물*.
- **의미(dense) 검색**: 임베딩 벡터로 뜻이 비슷한 걸 찾는다(위의 HNSW).

이 둘을 합치는 표준 기법이 **RRF(Reciprocal Rank Fusion)**: 각 검색의 *순위*를 `1/(k+순위)`로 점수화해 더한다(k는 보통 60). 점수값이 아니라 *순위*만 쓰니 서로 다른 검색의 스케일 차이를 안 맞춰도 돼서 깔끔하다. → 키워드가 놓친 건 의미가, 의미가 놓친 건 키워드가 건지는 상호보완.

### 5. 인덱스를 *어디에* 두느냐가 속도를 가른다

검색 품질만큼 중요한 게 *지연(latency)*인데, 의외로 알고리즘이 아니라 **인덱스가 어디 사는지**에서 갈리는 경우가 많다.

- **네트워크 파일시스템(NFS 등) 위의 임베디드 DB**(예: 파일 기반 SQLite): DB가 인덱스 페이지를 *잘게 여러 번 랜덤하게* 읽는데, 그 작은 읽기 하나하나가 **네트워크 왕복**이 된다. 게다가 색인을 주기적으로 재빌드하면 그 쓰기 I/O가 검색 읽기와 **같은 디스크 대역폭을 두고 경쟁**해서, 평균은 멀쩡해도 *꼬리(p95/p99)가 폭발*한다.
- **전용 DB 서버**(로컬 SSD + RAM 캐시 + MVCC): 쿼리를 서버에서 다 처리해 결과만 한 번에 주고, 자주 쓰는 데이터는 메모리에 캐시되며, 읽기가 쓰기에 막히지 않는다.

교훈: *같은 인덱스라도* 네트워크 너머 파일에 두면 느리고 전용 DB에 두면 빠르다. 성능 문제를 만나면 알고리즘 전에 **"이 데이터가 물리적으로 어디서 읽히나"**부터 의심하자. (I/O = 입출력, 디스크·네트워크와 데이터를 주고받는 일 — 보통 계산보다 이게 느리다.)

### 6. (가장 큰 교훈) 레버보다 eval이 먼저다

reranking이든 ef_search든, **전후 비교가 안 되면 다 추측**이다. 그래서 진짜 1순위는 개선 레버가 아니라 **답의 정확도를 점수화하는 평가(eval) 시스템**을 먼저 만드는 것.

특히 조심할 함정: **효율만 재고 정확도를 안 재는 것.** 검색 시스템 실험에서 "얼마나 빨리/적게 뒤졌나"(효율)만 측정하면, *빠르지만 틀린 답*을 좋다고 결론낼 수 있다. 지식검색의 북극성은 효율이 아니라 **답이 맞았나(정확도) + 근거가 진짜인가(충실도)**다.

eval을 만들 때 지표를 2층으로 나누면 깔끔하다:

- **검색 지표:** Recall@k(정답 문서가 상위 k에 들어왔나), MRR(평균 몇 등) → ef_search/recall 문제인지 판별
- **답변 지표:** 정확도, 충실도(환각 여부), 출처 정확도 → reranking 효과 측정

### 7. (보너스) 한 장면으로 단정하지 말고 추세로 교차검증

성능 분석하다 얻은 교훈. 로그 한 컷이나 한 번의 관찰로 원인을 단정하면 오진하기 쉽다. 평균만 봐도, 한 장면만 봐도 둘 다 틀릴 수 있다. **정규식·집계로 추세(평균·p95·분포)를 보고 교차검증한 뒤** 결론을 내야 한다. 꼬리(p95/max)가 평균과 *전혀 다른* 범인을 가리키는 경우가 많다.

---

**한 줄 요약:** HNSW는 벡터 검색을 빠르게 하는 다층 지도, ef_search는 그 정확도↔속도 다이얼, reranking은 순서를 바로잡는 레버 — 하지만 이 모든 걸 *측정할 eval*이 없으면 전부 추측이다. 개선보다 평가가 먼저.
